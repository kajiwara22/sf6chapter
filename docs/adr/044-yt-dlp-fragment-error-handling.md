# ADR-044: yt-dlp フラグメントエラー発生時のダウンロード中断

## ステータス

承認済み - 2026-06-08

## 文脈

### 発端

yt-dlp によるダウンロード中に以下のようなエラーが発生していることがログから確認された。

```
[download] Got error: HTTPSConnectionPool(host='rr1---sn-4g5lznls.googlevideo.com', port=443): Read timed out. (read timeout=20.0)
[download] fragment not found; Skipping fragment 380 ...
```

このエラーはダウンロードファイル（HLS ストリーム）の一部フラグメントが取得できなかったことを示す。フラグメントが欠損した状態でも後続処理（フレーム抽出・テンプレートマッチング・Gemini 認識）が続行されると、YouTube 上の動画とローカルファイルの内容にずれが生じ、チャプターのタイムスタンプが不正確になる。

あわせて、yt-dlp のメッセージが Python の `logging` を経由せず直接 stdout へ出力されていたため、ログファイルにこれらのエラーが記録されていなかった問題も確認された。

### yt-dlp のログ出力問題

yt-dlp はデフォルトで `[download]` や `[hlsnative]` などの進捗・エラーメッセージを直接 stdout に書き出す。`logging` モジュールを介さないため、ファイルハンドラに届かずログファイルに残らなかった。

`ydl_opts` に `logger` キーでカスタムオブジェクトを渡すことで、yt-dlp の全出力を Python logging 経由にリダイレクトできる。これは ADR-044 の実装として合わせて対応する。

### `ignoreerrors: True` の設定について

`VideoDownloader` には初期実装から `ignoreerrors: True` が設定されていた。特定のユースケース（一時的なネットワーク瞬断を無視したいなど）を意図した形跡はなく、初期実装時にデフォルト的に設定されたものと判断した。

この設定があるために yt-dlp はフラグメント取得に失敗しても処理を継続し、欠損したままダウンロードを完了扱いにしていた。

### 対応の選択肢

| 案 | 内容 | 評価 |
|---|---|---|
| A | `ignoreerrors: False` に変更し、yt-dlp 自体にエラーを投げさせる | シンプル。ただしフラグメントエラー以外の軽微なエラーでも止まるようになる可能性がある。また `ignoreerrors` を `True` にした意図が不明な以上、変更の影響範囲が不明確 |
| B | `_YtDlpLogger` にエラー検知フラグを持たせ、ダウンロード後にチェックして例外を投げる | フラグメントエラーのみを選択的に検知できる。ただし Logger にフラグ管理という副作用を持たせることになり、単一責務の観点から設計として不適切 |
| C | `ignoreerrors: True` を維持しつつ、yt-dlp の `progress_hooks` を使ってフラグメントエラーを検知する | Logger の責務を汚染せず、かつ `ignoreerrors` の設定を変えずにフラグメントエラーだけを捕捉できる |

方案 B は「Logger がエラー状態を追跡する」という責務の混在が問題であり、設計上好ましくない。方案 A はシンプルだが `ignoreerrors` の変更影響が不明確。方案 C が責務分離と既存設定の維持を両立できる。

## 決定事項

**方案 C を採用する。** `ignoreerrors: True` を維持したまま、yt-dlp の `progress_hooks` でフラグメントエラーを検知し、`download()` の呼び出し元に例外を伝播させる。

あわせて、yt-dlp のログ出力を Python logging 経由にするため `_YtDlpLogger` を導入する（ログファイルへの記録漏れ対策）。

### 実装方針

- `ydl_opts` に `progress_hooks` を追加し、`status == "error"` かつメッセージに `fragment` を含む場合にフラグを立てる
- `ydl.download()` 完了後にフラグを確認し、`True` であれば `RuntimeError` を投げる
- フラグ管理は `VideoDownloader.download()` のローカルスコープに閉じ、Logger には持たせない

## 結果

### 良い点

- フラグメント欠損のままチャプター処理が進むことを防ぎ、タイムスタンプのずれを未然に回避できる
- `ignoreerrors` の設定を変更しないため、既存の動作への影響を最小限に抑えられる
- Logger の責務をログ出力のみに保てる
- yt-dlp の全出力がログファイルに記録されるようになり、障害調査が容易になる

### 制約・トレードオフ

- `progress_hooks` の `status` や渡されるオブジェクトの仕様は yt-dlp の内部実装に依存するため、バージョンアップで変わる可能性がある
- フラグメントエラーが発生した動画は処理がスキップされるため、再試行は手動（または別途リトライ機構の実装）が必要
