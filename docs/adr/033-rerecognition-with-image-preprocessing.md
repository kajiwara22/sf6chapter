# ADR-033: Battlelog未マッチ時の画像前処理による再認識

## ステータス

提案（Proposed） - 2026-03-20

## 文脈

### 問題の概要

Gemini APIによるキャラクター認識（ADR-019で改善済み）において、特定のステージ背景で文字色と背景色が近い場合に認識精度が大幅に低下する問題が確認されている。

### 具体的な事例

**Video ID `aUZZB4Jt-h0`** での認識失敗箇所:
- 227s: 文字と背景が近い色で認識失敗
- 2152s: 同上
- 2292s: 同上

これらはすべて同一ステージ（背景色が文字色に近い）での対戦であり、他のVideo IDでも同様の傾向が確認されている。

### 現状の認識フロー

```
フレーム画像（前処理なし）
  → Gemini API（OCRプロンプト + temperature=0.1 + enum制約）
  → キャラクター名正規化
  → チャプター生成
  → Battlelogマッチング
  → マッチ失敗 → 手動修正が必要
```

### 根本原因の分析

- ADR-019のプロンプト改善はJP/JAMIE等の「類似名混同」には効果的だった
- しかし**画像自体の視認性が低い**ケースには対応できない
- 現在Gemini APIへの入力フレームはRGB変換のみで、画像前処理は行われていない
- テンプレートマッチング用にはエッジ抽出（グレースケール→ガウシアンブラー→Canny）が実装されているが、認識用には未適用

### 影響範囲

- 認識誤りによるBattlelog未マッチ分は、現在手動でchapters.jsonを修正して対応している
- 動画あたり推定3-4件の手動修正が発生

## 決定

Battlelogマッチング後に、**未マッチかつキャラクター名が認識済み（UNKNOWN_CHARACTER以外）のチャプター**に対して、フレーム画像に前処理を適用した上でGemini APIに再認識を依頼し、再度Battlelogマッチングを試みる。

### 処理フロー

```
[既存フロー]
検出 → 認識 → チャプター生成 → Battlelogマッチング
                                        │
                                        ▼
                              matched=True → 完了
                              matched=False かつ UNKNOWN → 完了（対象外）
                              matched=False かつ 認識済み → 再認識ループへ
                                        │
                                        ▼
                              [新規: 再認識ループ]
                              中間ファイルからフレーム画像を読み込み
                                        │
                                        ▼
                              画像前処理（手法は事前検証で決定）
                                        │
                                        ▼
                              Gemini API 再認識
                                        │
                                        ▼
                              認識結果が元と異なる → Battlelog再マッチ
                              認識結果が元と同じ → 完了（改善なし）
                                        │
                                        ▼
                              再マッチ成功 → チャプター更新
                              再マッチ失敗 → 完了（元のまま）
```

### トリガー条件

以下の**すべて**を満たすチャプターが再認識の対象:

1. `matched == False`（Battlelogマッチング失敗）
2. チャプタータイトルに有効なキャラクター名が含まれている（`UNKNOWN_CHARACTER`でない）
3. 対応するフレーム画像が中間ファイルに存在する

### 再認識の回数

**1回のみ**。前処理手法は事前検証で最も効果的と判断されたものを1つ適用する。

### 前処理手法の候補

事前検証で以下の手法を比較し、最適なものを選定する:

| 手法 | 概要 | 期待される効果 |
|------|------|---------------|
| CLAHE | 局所的コントラスト強調 | 背景と文字のコントラスト改善 |
| Adaptive Threshold | 局所的な二値化 | 文字を白黒で明確化 |
| Unsharp Mask | エッジ強調 | 文字の輪郭を際立たせる |
| LAB色空間のL成分CLAHE | 輝度チャネルのみ処理 | 色情報を保持しつつコントラスト改善 |
| ネガポジ反転 | 画像全体の反転 | 背景と文字の関係を逆転 |

**事前検証方法**: 認識失敗が確認されている `aUZZB4Jt-h0` の227s, 2152s, 2292sのフレーム画像を使用し、各手法で前処理後にGemini APIで認識精度を比較する。

### 再認識結果の判定

1. 前処理後のGemini API認識結果を取得
2. 元の認識結果と異なる場合のみ、Battlelog再マッチングを実行
3. 再マッチ成功（`matched == True`）の場合、チャプター情報を更新
4. 再マッチ失敗または元と同じ結果の場合、元のデータを維持

### 中間ファイルへの記録

再認識を実施したチャプターについて、`chapters.json`に以下のフィールドを追加:

```json
{
  "startTime": 227,
  "title": "GOUKI VS JP",
  "matched": true,
  "rerecognized": true,
  "original_title": "GOUKI VS JAMIE",
  "rerecognition_method": "clahe",
  "confidence": "high",
  "player1_character": "gouki",
  "player2_character": "jp",
  "replay_id": "XXXXX"
}
```

- `rerecognized`: 再認識が実施された事実を記録
- `original_title`: 前処理前の認識結果（効果測定用）
- `rerecognition_method`: 適用した前処理手法名

**注**: R2に格納するParquet（matches.parquet等）には再認識メタデータを含めない。Parquetは最終結果のみを保持し、再認識の経緯はデバッグ・効果検証用途として中間ファイルに限定する。

## 検討した代替案

### 代替案1: 全フレームに前処理を適用

すべてのフレーム画像に前処理を適用してからGemini APIに送信する。

**却下理由**:
- 元々コントラストが十分なフレームに前処理を適用すると逆効果の可能性がある
- Gemini APIの認識能力は大半のフレームで十分機能している
- 全フレーム分のAPI呼び出しが追加で発生し、コスト・時間が増加

### 代替案2: 画像品質スコアによる事前判定

フレーム画像のコントラスト比や文字領域のヒストグラムを分析し、低品質と判定されたフレームのみ前処理を適用する。

**却下理由**:
- 品質スコアの閾値設計が困難（何をもって「低品質」とするか）
- Battlelog未マッチをトリガーとする方が確実で実装がシンプル
- 将来の改善で品質スコアを組み合わせることは可能

### 代替案3: 複数手法の前処理を順番に試行

再認識が成功するまで複数の前処理手法を順番に適用する。

**却下理由**:
- Gemini API呼び出しが手法数分増加（コスト・時間）
- 事前検証で最適手法を1つ選定する方が効率的
- 1手法で改善しないケースは根本的に画像品質の問題であり、手法を変えても改善する可能性が低い

### 代替案4: Battlelogマッチング不使用時のフォールバック

Battlelogデータが取得できない環境でも動作する前処理パイプラインを構築する。

**却下理由**:
- 現状Battlelogなしのケースは運用上考慮不要
- トリガー条件の判定にBattlelogマッチング結果が必要
- 将来必要になった場合は別途対応可能

## 成功基準

| 指標 | 現状（推定） | 目標 |
|------|-------------|------|
| Battlelog未マッチ率（認識誤り起因） | 動画あたり3-4件 | 動画あたり1-2件 |
| 前処理による再認識成功率 | N/A | 50%以上 |
| 手動修正が必要なチャプター数 | 動画あたり3-4件 | 動画あたり1-2件（50%削減） |
| 処理時間増加 | N/A | 1動画あたり+30秒以内 |
| 既存認識精度への影響 | N/A | 劣化なし（再認識は未マッチ分のみ） |

## リスクと対策

| リスク | 影響度 | 対策 |
|-------|--------|------|
| 事前検証でどの前処理手法も効果なし | 高 | ADR自体を保留とし、モデルアップグレード等の別アプローチを検討 |
| 前処理で元々正しかった認識が悪化 | 中 | 再認識結果はBattlelog再マッチが成功した場合のみ採用。失敗時は元データを維持 |
| Gemini API追加呼び出しのコスト | 低 | 未マッチ分（動画あたり3-4件）のみ。gemini-2.5-flash-liteのコストは低い |
| 中間ファイルにフレーム画像がないケース | 低 | トリガー条件でフレーム画像の存在を確認。なければスキップ |

## 実装計画

### Phase 1: 事前検証（Notebook）

1. 認識失敗フレーム画像を収集（`aUZZB4Jt-h0` + 他Video ID）
2. 各前処理手法を適用した画像を生成
3. Gemini APIで認識精度を比較
4. 最適な前処理手法を選定

### Phase 2: パイプライン統合

1. `packages/local/src/character/recognizer.py` に前処理付き再認識メソッドを追加
2. `packages/local/main.py` のBattlelogマッチング後に再認識ループを挿入
3. `chapters.json` への再認識メタデータ記録を実装

### Phase 3: 効果測定

1. 複数動画で再認識の効果を検証
2. 成功基準を満たしているか評価
3. 必要に応じてパラメータ調整
