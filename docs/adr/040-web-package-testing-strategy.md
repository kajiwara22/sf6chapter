# ADR-040: packages/web のテスト戦略

## ステータス

採用 - 2026-05-05

## 文脈

### 現状

`packages/web` には現時点でテストが存在しない。ビルドと型チェックのみを品質担保として運用している。

```
packages/web/
├── src/
│   ├── client/
│   │   ├── search.ts         # DuckDB-WASMクエリ・JST/UTC変換ロジック
│   │   ├── components/       # UI コンポーネント
│   │   └── main.ts
│   ├── server/
│   │   └── routes/api.ts     # Hono APIルート（Presigned URL生成）
│   └── shared/types.ts
└── package.json              # ビルド・型チェックスクリプトのみ
```

### 課題

E2E テストの整備コストは高く、外部依存（DuckDB-WASM、Cloudflare R2、実データ）が多いため現時点では採算が合わない。一方で、ロジックのバグを検出する手段がまったく存在しない状態は避けたい。

特に `search.ts` の JST/UTC 変換関数（`convertJstDateToUtc`、`convertJstDateTimeToTimestamp`）は以下の特性を持つ：

- **影響範囲が広い**: 全フィルター機能（期間指定）の正確性がこの関数に依存する
- **バグが無症状**: 動作はするが期間フィルターが意図しない範囲になるサイレントバグが起きやすい
- **純粋関数**: 外部依存ゼロでテストが書きやすい

## 選択肢と結論

**採用**: ビルド・型チェックのみを CI で担保しつつ、JST/UTC 変換関数のユニットテストを追加する。

## 各選択肢の比較表

| 選択肢 | コスト | 効果 | 採用 |
|--------|--------|------|------|
| ビルド + 型チェックのみ（現状） | 低 | 型の整合性のみ | - |
| JST/UTC 変換関数のユニットテスト追加 | 低 | 最も壊れやすいロジックを担保 | ✅ |
| Hono APIルートの単体テスト | 中 | 薄いラッパーのため効果が限定的 | ✗ |
| DuckDB クエリのテスト | 高 | ブラウザ WASM 前提で環境構築が困難 | ✗ |
| E2E テスト（Playwright） | 高 | 実際の動作を担保できるが整備コストが大きい | 別課題 |

## 結論を導いた重要な観点

### E2E テストは別課題とする

実際のブラウザでの動作確認はユーザーが行う。E2E テストの整備は独立した課題として将来対応する。

### ユニットテスト対象の絞り込み

テスト対象を「外部依存ゼロの純粋関数」に限定する。DuckDB-WASM を必要とするクエリ関数や、Cloudflare R2 との通信を伴う API ルートはテスト環境の構築コストが高いため除外する。

### 担保する範囲

| 項目 | 手段 |
|------|------|
| 型の整合性 | `tsc --noEmit` |
| ビルドの正しさ | `vite build` |
| JST/UTC 変換ロジック | Vitest ユニットテスト |
| ブラウザ動作 | ユーザーによる手動確認 |

## 帰結

### メリット

- 期間フィルターの根幹となる変換ロジックが自動検証される
- ツール追加（Vitest）のコストが低い
- CI パイプラインの実行時間への影響が軽微

### トレードオフ

- DuckDB クエリのロジック（フィルター条件の組み合わせ、FULL OUTER JOIN の挙動など）は担保されない
- UIコンポーネントの挙動は担保されない

### 将来の見直し条件

- E2E テスト整備を正式に着手する場合は、このADRを更新して Playwright 等の方針を追記する
- 新たに外部依存のない純粋関数が追加された場合は、同様にユニットテストを追加する
